# 代码库深度剖析 — LLM Prompt 模板

> 指导 LLM 系统性地阅读一个代码库并产出经过验证的架构文档。
> 核心价值在于**阅读方法论**——怎么读、按什么顺序读、每层读到什么粒度。

---

## 使用方法

将 `{{变量}}` 替换为实际值，发送给 LLM（推荐 Claude Opus + Agent 能力）。

| 变量 | 示例 |
|------|------|
| `{{项目名}}` | Claude Code |
| `{{代码库路径}}` | `/home/user/my-project` |
| `{{语言}}` | TypeScript / Python / Go |
| `{{项目简介}}` | AI 编程助手 CLI 工具 |
| `{{输出路径}}` | `docs/deep-dive/` |

---

## Prompt

````markdown
# 任务：深度阅读 {{项目名}} 代码库并产出架构文档

代码库路径：{{代码库路径}}
语言：{{语言}}
简介：{{项目简介}}
输出目录：{{输出路径}}

---

## 第一部分：怎么读代码

整个阅读过程分为四层，从粗到细，每一层都有明确的目标和停止条件。
**不要跳层**——没有完成上一层就深入下一层，会迷失在细节里。

### 第一层：鸟瞰（30 分钟）

**目标**：建立全局心智模型——这个系统有几个部分，它们怎么连接。

**操作**：
```bash
# 1. 规模感知
find {{代码库路径}} -type f -name "*.{{ext}}" | wc -l        # 多少个文件
find {{代码库路径}} -type f -name "*.{{ext}}" -exec wc -l {} + | sort -rn | head -20  # 最大的 20 个文件
ls {{代码库路径}}/src/                                         # 顶层目录结构（或等价的源码根目录）

# 2. 入口识别
grep -r "main\|entrypoint\|bootstrap" {{代码库路径}}/package.json  # 或 go.mod / setup.py / Cargo.toml
```

**阅读顺序**：
1. **README / CLAUDE.md / 架构文档**（如果有）——获取作者意图
2. **包管理文件**（package.json / go.mod / pyproject.toml）——依赖关系揭示技术选型
3. **入口文件**（main / cli / app）——只读前 50 行，理解启动做了什么
4. **顶层目录**——每个目录扫一眼 index/mod 文件，理解职责划分

**这一层要回答**：
- 系统有哪些主要模块？（列出 5-10 个）
- 数据从哪里进来，从哪里出去？
- 核心循环是什么？（请求-响应？事件循环？管道？）

**停止条件**：能画出一张 5-10 个框的架构草图。

---

### 第二层：主干追踪（1-2 小时）

**目标**：理解核心数据流——一个请求/命令从进入到完成经过了哪些步骤。

**方法：选一条主路径，端到端追踪。**

例如对 CLI 工具：用户输入一个命令 → 解析 → 处理 → 调用外部服务 → 返回结果。
例如对 Web 服务：HTTP 请求 → 路由 → 中间件 → handler → 数据库 → 响应。

**操作**：
1. 从入口文件开始，跟随 `import`/`require`/调用链往下走
2. **每个函数只看签名和前 10 行**，理解它做什么、调用了谁，然后跳到下一个
3. 遇到分支（if/switch），只走主路径，标记其他分支留待第三层
4. 记录调用链：`A.func1() → B.func2() → C.func3()`

**关键技巧**：
- **跟着类型走**：函数参数和返回值的类型定义揭示数据结构
- **跟着错误走**：catch/error handling 暴露了系统认为什么会出错
- **跟着配置走**：哪些行为是可配置的 = 哪些是设计者认为会变化的

**这一层要回答**：
- 核心流程有几步？每步做什么？
- 关键数据结构是什么？（messages, state, config...）
- 哪些是"自己写的"，哪些是"调库的"？

**停止条件**：能描述出"一个请求从头到尾经过了哪些函数"。

---

### 第三层：子系统深潜（每个子系统 30-60 分钟）

**目标**：理解每个子系统的内部工作原理。

**方法：对第一层识别的每个模块，执行"三步读法"。**

**Step 1 — 接口层**（10 分钟）
- 读该模块的公开 API（export 了什么函数/类？）
- 读类型定义（interface / type / struct）
- 这告诉你"这个模块对外承诺了什么"

**Step 2 — 核心逻辑**（20-30 分钟）
- 找到该模块最大的 1-3 个文件，按行数排序
- 只读核心函数（最长的、被调用最多的）
- 关注：状态机、循环、条件分支、错误处理

**Step 3 — 边界和约束**（10 分钟）
- 这个模块和其他模块怎么通信？（函数调用？事件？消息队列？）
- 有什么安全/权限检查？
- 有什么配置项控制行为？

**怎么判断哪个子系统先读？按依赖关系排序：**
- 被依赖最多的先读（通常是：类型定义 → 核心逻辑 → 工具/插件 → UI）
- 如果不确定：`grep -r "import.*from.*模块名" src/ | wc -l` 看谁被引用最多

**这一层要回答**（每个子系统）：
- 它的核心抽象是什么？（Tool? Hook? Provider? Message?）
- 有几种实现/变体？
- 关键常量和阈值是什么？

**停止条件**：能解释每个子系统"为什么这样设计"。

---

### 第四层：细节验证（按需）

**目标**：精确获取具体数字和实现细节。

**这一层不是"通读"，而是"查字典"——带着问题去读。**

**常见需要验证的细节**：
```bash
# 某个类型有多少字段？
grep -A 50 "export type Tool" src/Tool.ts | head -60

# 某个常量的值？
grep -rn "MAX_SNAPSHOTS\|CACHE_SIZE\|TIMEOUT" src/

# 某个函数的行数？
wc -l src/query.ts

# 某个目录有多少文件？
find src/tools/ -type d | wc -l

# 某个函数被谁调用？
grep -rn "functionName(" src/
```

**这一层的原则**：
- **不要猜**——不确定就 grep
- **不要估**——行数、文件数、字段数都用命令实测
- **不要推断行为**——读代码确认，不要从函数名推断

## 第二部分：怎么切中重点

### 识别"重要"的信号

不是所有代码都值得同等关注。以下信号帮助判断优先级：

| 信号 | 含义 | 怎么发现 |
|------|------|---------|
| 文件行数 Top 10 | 复杂度集中处 | `wc -l` 排序 |
| 被 import 最多 | 核心依赖 | `grep -c "from.*模块"` |
| 类型定义文件 | 系统契约 | `**/types.ts`, `**/types.go` |
| 常量/配置文件 | 设计决策的物化 | `constants.ts`, `config.ts` |
| 测试文件 | 暴露预期行为 | `**/*.test.*`, `**/*_test.go` |
| 注释密度高的函数 | 作者认为需要解释的地方 | 扫读时注意 |
| Feature flag 门控 | 演进中的功能 | `grep "feature\|flag\|gate"` |

### 避免的陷阱

1. **不要从测试开始读**——测试依赖对系统的理解，先理解系统再看测试
2. **不要在第一层就深入工具函数/utils**——它们是叶子节点，最后读
3. **不要按字母序读文件**——按依赖序和调用链读
4. **不要试图理解每一行**——第三层之前，函数只看签名和前 10 行
5. **不要相信注释和文档超过代码**——代码是 ground truth，注释可能过期

### 不同类型项目的重点差异

| 项目类型 | 先读什么 | 核心要理解什么 |
|----------|---------|--------------|
| CLI 工具 | 入口 → 命令注册 → 核心命令实现 | 参数解析、子命令分发、输出格式 |
| Web 服务 | 路由表 → 中间件链 → 数据模型 | 请求生命周期、认证授权、数据流 |
| SDK/库 | 公开 API → 类型定义 → 内部实现 | 抽象层次、扩展点、错误处理 |
| 编译器/解释器 | 语法定义 → 解析器 → IR → 后端 | 各阶段的数据结构变换 |
| Agent 系统 | 核心循环 → 工具系统 → 上下文管理 | 决策流程、工具调度、状态管理 |

## 第三部分：产出文档

阅读完成后，将理解转化为文档。

### 章节规划

按**读者心智模型**组织（不是代码目录结构）：

```
全局概览（系统是什么） → 核心流程（怎么工作） → 子系统详解（每部分怎么实现）
→ 扩展机制（怎么定制） → 基础设施（怎么构建和部署） → 设计哲学（为什么这样做）
```

每章 3000-8000 字，10-20 章（视项目规模）。强相关模块合并为一章。

### 每章结构

1. 一句话概括
2. 架构图（Mermaid，`%%{init: {'theme': 'neutral'}}%%`）
3. 核心概念和数据结构
4. 流程描述（配时序图或流程图）
5. 关键实现细节（带代码行号引用）
6. 本章关键文件表（文件 + 行数 + 职责）

### 附加产出

- **README.md**：章节目录 + 阅读路径（初学者/快速查阅/开发者）
- **附录**：关键文件索引 + 术语表 + 环境变量速查

---

## 第四部分：验证

**写完不等于写对。** 每一条事实性声明都需要代码验证。

### 校准流程

对每章逐段提取事实性声明，在代码中验证：
- ✅ 正确 → 保留
- ❌ 错误 → 替换为代码验证值
- ⚠️ 偏差 → 更新为精确表述
- ➖ 无法验证 → 从文档删除

### 验证命令速查

```bash
wc -l file.ts                          # 行数
grep -c "pattern" file.ts              # 计数
grep -rn "functionName" src/           # 定位函数
find src/ -name "*.ts" -type f | wc -l # 文件数
test -f path/to/file && echo "exists"  # 路径存在
```

### 通过标准

- 0 个❌（错误事实）
- 0 个➖（无法验证的声明）
- 所有 Mermaid 图节点名与代码中的实际名称一致
````
