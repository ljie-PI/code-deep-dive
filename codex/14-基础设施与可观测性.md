# 第14章 基础设施与可观测性

Codex 的基础设施层覆盖六个独立子系统：`analytics` 负责产品分析事件上报、`otel` 提供 OpenTelemetry 可观测性（日志/追踪/指标三支柱）、`models-manager` 管理模型目录与缓存、`features` 实现功能标志系统、`apply-patch` 处理自定义补丁格式、`file-search` 提供模糊文件搜索、`connectors` 管理第三方应用连接器。

---

## 14.1 Analytics：产品分析事件

### 14.1.1 架构概览

`analytics/` crate 实现了异步、带背压的分析事件管道。核心组件 `AnalyticsEventsClient` 通过 mpsc channel 解耦事件生产与消费：

```
事件产生 → AnalyticsEventsClient → mpsc channel(256) → AnalyticsReducer → HTTP 上报
```

**AnalyticsEventsQueue** 封装了 channel 的发送端和去重状态：

```rust
pub(crate) struct AnalyticsEventsQueue {
    pub sender: mpsc::Sender<AnalyticsFact>,
    pub app_used_emitted_keys: Arc<Mutex<HashSet<(String, String)>>>,
    pub plugin_used_emitted_keys: Arc<Mutex<HashSet<(String, String)>>>,
}
```

关键参数：
- `ANALYTICS_EVENTS_QUEUE_SIZE = 256`：channel 容量，满时新事件将背压
- `ANALYTICS_EVENTS_TIMEOUT = 10s`：上报超时
- `ANALYTICS_EVENT_DEDUPE_MAX_KEYS = 4096`：去重集合最大容量

### 14.1.2 事件去重

为避免高频事件的重复上报，`AnalyticsEventsClient` 维护两个去重集合：

- `app_used_emitted_keys`：`(app_id, action)` 对去重，用于 `codex_app_used` 事件
- `plugin_used_emitted_keys`：`(plugin_id, action)` 对去重，用于 `codex_plugin_used` 事件

每个集合最多容纳 4096 个键。超过限制后不再去重（允许重复上报），以避免无限增长的内存开销。

### 14.1.3 AnalyticsReducer

`AnalyticsReducer`（`reducer.rs`）是事件管道的消费端，负责将原始的 `AnalyticsFact` 聚合为可上报的 `TrackEventRequest`。它从 mpsc channel 接收事件，批量组装为 `TrackEventsRequest`，通过 HTTP POST 发送到分析后端。

### 14.1.4 事件类型

`events.rs` 定义了完整的事件体系，通过 `TrackEventRequest` 结构上报：

| 事件名称 | 说明 |
|----------|------|
| `codex_thread_initialized` | 线程初始化 |
| `codex_turn_event` | Turn 级别事件（开始/完成/中断） |
| `codex_compaction_event` | 上下文压缩事件 |
| `codex_app_mentioned` | 应用在对话中被提及 |
| `codex_app_used` | 应用被实际调用（去重） |
| `codex_plugin_*` | 插件相关事件（安装/卸载/状态变更） |
| `guardian_review` | Guardian 自动审查事件 |
| `skill_invocation` | Skill 调用事件 |

事件元数据通过 `current_runtime_metadata()` 获取运行时上下文，`AppServerRpcTransport` 枚举区分传输方式。

### 14.1.5 Facts 体系

`facts.rs` 定义了类型安全的事件输入（"facts"），每种 fact 对应一个具体的业务动作：

- `AppInvocation` / `AppMentionedInput` / `AppUsedInput`：应用相关
- `PluginState` / `PluginStateChangedInput`：插件状态
- `SkillInvocation` / `SkillInvokedInput`：Skill 调用
- `SubAgentThreadStartedInput`：子 Agent 线程启动
- `TurnResolvedConfigFact` / `TurnTokenUsageFact`：Turn 级别配置与 Token 用量
- `CustomAnalyticsFact`：自定义扩展事件
- `TrackEventsContext`：上报上下文

## 14.2 OpenTelemetry：可观测性三支柱

### 14.2.1 OtelProvider

`otel/` crate 基于 OpenTelemetry SDK 提供日志（Logs）、追踪（Traces）和指标（Metrics）三支柱可观测性。入口类型 `OtelProvider` 统一管理三个 provider：

```rust
pub struct OtelProvider {
    pub logger: Option<SdkLoggerProvider>,
    pub tracer_provider: Option<SdkTracerProvider>,
    pub tracer: Option<Tracer>,
    pub metrics: Option<MetricsClient>,
}
```

`OtelProvider::from(settings)` 根据 `OtelSettings` 配置初始化，支持三种导出器：
- `OtelExporter::None`：禁用
- `OtelExporter::OtlpGrpc`：gRPC 导出（支持 TLS）
- `OtelExporter::OtlpHttp`：HTTP 导出（Binary/JSON 协议）

每个 provider 独立配置，可以单独启用日志、追踪或指标。Resource 属性包含 service name、version、environment 和可选的 host name（仅日志）。

### 14.2.2 SessionTelemetry

`SessionTelemetry`（`events/session_telemetry.rs`，1097 行）是会话级遥测的核心协调器，提供结构化的事件记录方法：

**API 请求追踪**：
- `record_api_request()`：记录 API 请求延迟、状态、模型等 → 指标 `codex.api_request`
- `record_responses()`：记录响应流事件

**WebSocket 追踪**：
- `record_websocket_connect()`：连接事件 → `codex.websocket_connect`
- `record_websocket_request()`：请求事件 → `codex.websocket_request`
- `record_websocket_event()`：流事件 → `codex.websocket_event`

**SSE 追踪**：
- SSE 事件记录 → `codex.sse_event`

**工具调用追踪**：
- 工具决策/结果记录 → `codex.tool_result`
- 持续时间 → `codex.tool.call`

**Turn 级指标**：
- `codex.turn.e2e_duration_ms`：端到端延迟
- `codex.turn.ttft.duration_ms`：首 Token 延迟

**线程追踪**：
- `codex.thread.started`：线程启动

所有指标方法通过 `record_duration()` 统一调用底层 `MetricsClient`，并带上 session-level 的标签（auth_mode、model、provider 等）。

### 14.2.3 W3C TraceContext 支持

`trace_context.rs` 实现了 W3C TraceContext 规范的注入/提取：

```rust
pub fn current_span_w3c_trace_context() -> Option<W3cTraceContext>;
pub fn context_from_w3c_trace_context(trace: &W3cTraceContext) -> Option<Context>;
pub fn set_parent_from_w3c_trace_context(span: &Span, trace: &W3cTraceContext) -> bool;
pub fn traceparent_context_from_env() -> Option<Context>;
```

支持通过 `TRACEPARENT` / `TRACESTATE` 环境变量传入父追踪上下文，实现跨进程追踪链路。`TRACEPARENT_CONTEXT` 使用 `OnceLock` 确保仅解析一次。

### 14.2.4 目标过滤

`targets.rs` 定义了日志和追踪的目标过滤逻辑：
- `is_log_export_target()`：匹配 `codex_otel.log_only` 和 `codex_otel.network_proxy` 等前缀
- `is_trace_safe_target()`：仅匹配 `codex_otel.trace_safe` 前缀

通过 `tracing_subscriber::filter::filter_fn` 将过滤器注入 tracing 订阅层，确保敏感数据不会泄露到追踪后端。

## 14.3 Models Manager：模型目录管理

### 14.3.1 核心职责

`models-manager/` crate 管理可用模型的发现、缓存和刷新。核心逻辑在 `manager.rs`（586 行）。

### 14.3.2 内置模型目录

`bundled_models_response()` 从编译时嵌入的 `models.json` 文件返回静态模型列表，作为网络不可用时的回退。

### 14.3.3 缓存机制

`ModelsCacheManager`（`cache.rs`，183 行）实现了本地文件缓存：

| 参数 | 值 | 说明 |
|------|-----|------|
| 缓存文件 | `models_cache.json` | 位于 codex_home 下 |
| TTL | 300 秒（5 分钟） | 缓存过期时间 |
| 刷新超时 | 5 秒 | 网络请求超时 |
| 端点 | `/models` | API 端点 |

缓存刷新流程：
1. 检查本地 `models_cache.json` 是否存在且未过期
2. 若有效则直接返回缓存内容
3. 若过期或不存在，使用 `RwLock` 获取写锁，通过 `timeout(5s)` 限制的 HTTP 请求获取最新模型列表
4. 若请求失败（使用 `TryLockError` 处理竞态），回退到缓存或内置列表

### 14.3.4 认证模式

通过 `AuthMode` 枚举区分不同的认证方式：
- `AuthMode::ApiKey`：直接 API Key 认证
- `AuthMode::Chatgpt`：ChatGPT 认证流程（包括 `ChatgptAuthTokens` 变体）

认证状态影响可用模型列表——ChatGPT 认证模式下可能返回不同的模型集合。

### 14.3.5 协作模式预设

`collaboration_mode_presets.rs` 管理协作模式（如 Plan、Default）的模型预设配置，`builtin_collaboration_mode_presets()` 返回内置预设。

### 14.4.2 Stage 生命周期

每个 Feature 关联一个 `Stage`，表示其成熟度：

```rust
pub enum Stage {
    UnderDevelopment,
    Experimental { name, menu_description, announcement },
    Stable,
    Deprecated,
    Removed,
}
```

`Experimental` 阶段的特性会出现在 TUI 的 `/experimental` 菜单中，包含用户可见的名称、描述和公告文本。`Removed` 阶段的标志保留以确保旧配置文件的向后兼容性。

### 14.4.3 Features::from_sources()

功能标志的最终值通过多源合并确定：

```
基础配置 (base config)
  ↓ 覆盖
配置文件 (profile)
  ↓ 覆盖
命令行/环境覆盖 (overrides)
  ↓
最终 Features 集合
```

`Features::from_sources()` 方法按优先级合并这些来源。合并过程中：
- **依赖归一化**：某些标志有隐含依赖（如启用 X 自动启用 Y）
- **Legacy 键兼容**：`legacy.rs` 模块处理旧版配置键到新版 Feature 枚举的映射
- **实验菜单**：遍历所有 `Experimental` 阶段的标志，构建 TUI 菜单选项

### 14.4.4 FeatureToml

`FeatureToml<T>` 泛型枚举支持在 TOML 配置中表达特性值：

```rust
pub enum FeatureToml<T> {
    Bool(bool),
    Config(T),
}
```

允许特性既可以是简单的开关（`true`/`false`），也可以携带配置数据（如 `MultiAgentV2ConfigToml`）。

### 14.4.5 Legacy 兼容

`legacy.rs`（125 行）维护了一个从旧版配置键到 Feature 枚举的映射表。`LegacyFeatureToggles` 结构体处理旧格式的解析，`legacy_feature_keys()` 返回所有已知的旧版键名。

## 14.5 Apply-Patch：自定义补丁格式

### 14.5.1 补丁格式

`apply-patch/` crate 实现了 Codex 特有的补丁格式解析和应用。该格式使用显式标记界定补丁边界：

```
*** Begin Patch
*** Add File: path/to/new_file.rs
+line 1
+line 2
*** Update File: path/to/existing.rs
@@ context line
-old line
+new line
*** Delete File: path/to/removed.rs
*** End Patch
```

支持三种 Hunk 类型：
- `AddFile`：创建新文件
- `DeleteFile`：删除文件
- `UpdateFile`：修改文件（支持 `Move to:` 重命名）

### 14.5.2 解析流水线

`parser.rs`（901 行）实现了宽松解析器（`PARSE_IN_STRICT_MODE = false`，为兼容 gpt-4.1 的输出）。标记常量包括 `*** Begin Patch`、`*** End Patch`、`*** Add File:`、`*** Delete File:`、`*** Update File:`、`*** Move to:`、`*** End of File`、`@@` 等。

### 14.5.3 应用流水线

`lib.rs` 中的 `apply_patch()` 函数通过 `ExecutorFileSystem` trait 抽象文件系统操作，支持沙箱环境下的补丁应用。`invocation.rs`（857 行）处理从命令行参数到补丁执行的完整调用路径，`maybe_parse_apply_patch_verified()` 负责验证调用合法性。

`seek_sequence.rs`（163 行）实现了模糊行匹配算法，用于在文件内容与补丁上下文不完全一致时定位正确的应用位置。

### 14.5.4 独立可执行文件

通过 `CODEX_CORE_APPLY_PATCH_ARG1 = "--codex-run-as-apply-patch"` 标志，Codex 主可执行文件可以自调用为 apply-patch 子进程。`standalone_executable.rs` 处理这种自调用模式。

## 14.6 File-Search：模糊文件搜索

### 14.6.1 架构

`file-search/` crate（`lib.rs` 1187 行）基于 `nucleo` 库实现高性能模糊文件匹配，采用双线程架构：

- **Walker 线程**：使用 `ignore::WalkBuilder` 遍历目录树，将文件路径注入 `nucleo::Injector`
- **Matcher 线程**：`nucleo::Nucleo` 引擎在后台持续匹配，返回排序后的结果

### 14.6.2 FileSearchSession

`FileSearchSession` 管理搜索会话的生命周期，支持：
- 增量查询更新（查询变更时无需重新遍历目录）
- `FileSearchSnapshot` 中间状态快照，包含当前匹配结果、已扫描文件数、遍历是否完成
- `FileSearchResults` 最终结果，包含 `FileMatch` 列表和总匹配数

每个 `FileMatch` 包含 score、路径、匹配类型（File/Directory）、根路径，以及可选的字符级匹配索引（用于 UI 高亮）。

### 14.6.3 CLI 入口

`cli.rs` 提供命令行接口，`main.rs` 作为独立可执行文件入口。

## 14.7 Connectors：第三方应用连接器

### 14.7.1 概述

`connectors/` crate（`lib.rs` 549 行）管理第三方应用连接器的发现与缓存。

### 14.7.2 缓存策略

连接器列表通过全局 `ALL_CONNECTORS_CACHE`（`LazyLock<StdMutex<Option<CachedAllConnectors>>>`）缓存，TTL 为 1 小时（`CONNECTORS_CACHE_TTL = 3600s`）。

`AllConnectorsCacheKey` 包含 `chatgpt_base_url`、`account_id`、`chatgpt_user_id`、`is_workspace_account` 四个维度，确保不同账户/环境的缓存隔离。

### 14.7.3 Directory API

`list_all_connectors_with_options()` 异步分页获取连接器目录：
- 接收泛型 `fetch_page` 闭包，支持不同的 HTTP 客户端
- `DirectoryListResponse` 包含 `apps` 列表和 `next_token` 分页标记
- `DirectoryApp` 映射为 `AppInfo`，包含 id、name、description、branding、logo、labels 等
